## Getting Started

### Prerequisites
- GitHub account
- Basic Git knowledge
- Familiarity with HTML, CSS, JavaScript (for code contributions)
- Understanding of healthcare administration (preferred)

### Fork and Clone
```bash

# Clone your fork
git clone https://github.com/YOUR-USERNAME/derrick-mbesa-portfolio.git
cd derrick-mbesa-portfolio

# Add upstream remote
git remote add upstream https://github.com/Mbesad123/derrick-mbesa-portfolio.git
```

## Types of Contributions

### 📝 Documentation
- Improve README or other documentation
- Fix typos or clarity issues
- Add missing information
- Create guides or tutorials

**How to contribute:**
1. Edit the relevant markdown file
2. Make improvements with clear explanation
3. Submit a pull request

### 💻 Code Improvements
- Bug fixes
- Performance optimizations
- Accessibility improvements
- New features (discuss first)

**How to contribute:**
1. Create a new branch: `git checkout -b feature/your-feature`
2. Make changes following guidelines below
3. Test thoroughly
4. Submit pull request with description

### 📚 Content
- Blog articles on healthcare administration
- Case studies
- FAQs and guides
- Tutorial content
- Resource materials

**How to contribute:**
1. Write content in markdown format
2. Place in appropriate folder (`docs/blog/`, `docs/case-studies/`, etc.)
3. Include relevant metadata (date, author, tags)
4. Submit pull request

### 🐛 Bug Reports
- Found an issue? Report it!
- Check existing issues first
- Include reproduction steps
- Provide browser/system info

**How to report:**
1. Go to Issues
2. Click "New Issue"
3. Fill out the issue template
4. Submit

### 💡 Feature Suggestions
- Have an idea for improvement?
- Want to see new functionality?
- Suggest better wording or design?

**How to suggest:**
1. Go to Issues
2. Click "New Issue"
3. Choose "Feature Request" template
4. Describe idea clearly
5. Submit

## Development Workflow

### 1. Create a Branch
```bash
# Update your local main
git fetch upstream
git checkout main
git merge upstream/main

# Create feature branch
git checkout -b feature/descriptive-name
```

**Branch naming conventions:**
- `feature/` - New features
- `fix/` - Bug fixes
- `docs/` - Documentation
- `content/` - Blog posts, case studies
- `improve/` - Improvements to existing features

### 2. Make Changes
```bash
# Make your changes
# Commit with meaningful messages
git add .
git commit -m "Add feature description"
```

**Commit message guidelines:**
- Be descriptive and concise
- Use present tense ("Add feature" not "Added feature")
- Include relevant issue number if applicable
- Examples:
  - `Add case study for medical coding project`
  - `Fix typo in README`
  - `Improve mobile responsiveness`
  - `Fixes #123 - Add Calendly integration`

### 3. Push and Create Pull Request
```bash
# Push to your fork
git push origin feature/your-feature

# Go to GitHub and create pull request
```

**Pull Request Guidelines:**
- Title: Clear, descriptive title
- Description: Explain what and why
- Link related issues
- Include screenshots if relevant
- Checklist: Mark items as completed

### 4. Review Process
- Maintainer reviews your PR
- Provide feedback if needed
- Make requested changes
- PR gets merged when approved

## Guidelines

### Code Style
- Use meaningful variable names
- Add comments for complex logic
- Keep functions focused and small
- Follow existing code patterns
- Use 4-space indentation

### Documentation Style
- Use clear, simple language
- Include examples where helpful
- Add relevant links and resources
- Follow markdown best practices
- Proof-read before submitting

### Healthcare Compliance
- Never include patient information
- Protect confidential data
- Follow HIPAA standards
- Respect privacy regulations
- Use anonymized examples

### Content Quality
- Accurate, well-researched information
- Professional tone
- Properly formatted
- Include sources/citations
- Spell-checked

### Testing
Before submitting:
- Test functionality thoroughly
- Check on mobile devices
- Verify links work
- Test in multiple browsers
- Validate HTML/CSS

## Checklist
- [ ] My changes follow style guidelines
- [ ] I've updated documentation

## Dos and Don'ts

### ✅ Do
- Ask questions if unsure
- Test your changes thoroughly
- Write clear commit messages
- Be respectful and professional
- Follow the Code of Conduct
- Keep PRs focused and manageable
- Update documentation
- Include relevant issue numbers

### ❌ Don't
- Submit large, unfocused PRs
- Include patient information
- Use offensive language
- Make unrelated changes
- Ignore feedback
- Force push to shared branches
- Commit sensitive information
- Remove other contributors' work without discussion
